# Step 6: Create PR

> **Skill available:** Use `/omb pr` to automate this step. Generates PR tracking codes, detects categories, validates checklist, and creates the PR via `gh` CLI (or saves as markdown fallback). PR template reference: `skills/braincrew-create-pr/reference.md` § 1.

Every completed task ends with a clean PR. Delegate git operations to `git-master`.

## Branch Naming

| Type | Pattern | Example |
|------|---------|---------|
| Feature | `feature/<name>` | `feature/add-auth-middleware` |
| Bug fix | `fix/<name>` | `fix/token-expiry-race` |
| Release | `release/<version>` | `release/0.3.0` |
| Hotfix | `hotfix/<name>` | `hotfix/redis-connection-leak` |

## Conventional Commits

Format: `type(scope): description`

Types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `ci`.

Every commit includes `Co-Authored-By: Braincrew(dev@brain-crew.com)` trailer.

For full trailer specification, see `.claude/rules/git/commit-conventions.md`.

## PR Template

The full PR template lives in `skills/braincrew-create-pr/reference.md` § 1 (STANDARD and LIGHT tiers). The `/omb pr` skill auto-fills it. Key sections:

- **Tracking** — PR code + PLAN code traceability chain
- **Category** — auto-detected from file patterns
- **Summary / Motivation / Approach** — derived from plan Context and Architecture Decisions
- **Changes** — organized by category with file paths
- **Plan Tasks** — maps plan tasks to deliverables with status
- **Test Plan** — tool/target/result table
- **Verification Evidence** — links to `.omb/verifications/`
- **Documentation** — per-doc-type status table
- **Breaking Changes** — migration steps if any
- **Checklist** — pre-review validation

## Prohibited Content

- [HARD] Never include "Generated with Claude Code" or any variation (e.g., "Generated with [Claude Code]", "Generated by Claude Code") in PR descriptions, PR bodies, or commit messages. The only attribution is the `Co-Authored-By: Braincrew(dev@brain-crew.com)` trailer in commits and the `<!-- oh-my-braincrew | PR: ... -->` HTML comment in PR bodies.

## PR Checklist (before requesting review)

- [ ] All tests pass (pytest + vitest + tsc).
- [ ] No hardcoded secrets or credentials.
- [ ] Documentation updated per `.claude/rules/05-documentation.md`.
- [ ] Commits follow conventional format with trailers.
- [ ] Branch rebased on latest main.
- [ ] No unrelated changes included.
- [ ] Verification evidence collected (step 4 completed).
- [ ] No "Generated with Claude Code" text in PR body or commits.

## Review Assignment

- Code review → `reviewer` agent.
- Security-sensitive changes → `security-reviewer` agent.
- Never self-approve — authoring and review are separate passes.

## Merge Strategy

- Default: squash merge to main.
- Exception: release branches use merge commit to preserve history.
