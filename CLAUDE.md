<workflow>
Every non-trivial task follows this 6-step sequence:

1. Plan — produce plan in `.omb/plans/`. See `.claude/rules/01-plan.md`.
2. Review Plan — critic/reviewer validates before execution. See `.claude/rules/02-review-plan.md`.
3. Execute (TDD) — define phases first, write tests, then implement. See `.claude/rules/03-execute-tdd.md`. Automate with `/omb exec`.
4. Verify — run tests, collect evidence. See `.claude/rules/04-verify.md`.
5. Documentation — update docs, README, ADRs. See `.claude/rules/05-documentation.md`.
6. Create PR — conventional commits, PR description. See `.claude/rules/06-create-pr.md`. Automate with `/omb pr`.

Workflow Tiers:
- TRIVIAL (<10 lines, 1 file, no behavior change): Steps 3-6 only, lightweight. No plan file, no critic review.
- STANDARD (most tasks): All 6 steps.
- EMERGENCY (production hotfix, user says "skip plan"): Steps 3→4→6 only. Document skipped steps in commit trailer: `Skipped: plan,review,docs | hotfix urgency`
Cross-cutting conventions: `.claude/rules/code-conventions.md`, `.claude/rules/language-settings.md`.

Pipeline state is stored in `.omb/sessions/` as JSON. Each step maps to an omb sub-skill (e.g., step `create-plan` → `omb-create-plan`).
</workflow>

<subagent_invocation>
## Subagent Invocation Patterns

Three patterns for invoking subagents, from flexible to guaranteed.
**Preference order:** Use Pattern 3 (Agent tool) for all multi-agent skill invocations. Patterns 1-2 are for single-agent orchestrator-level delegation only.

### 1. Natural Language (flexible, Claude decides)
Reference the agent name in the prompt. Claude decides whether to delegate:
```
Use the test-engineer agent to write unit tests for the auth module
Have the security-reviewer agent check the middleware changes
```
Best for: orchestrator-level delegation in CLAUDE.md rules where Claude should choose the best agent.

### 2. @-Mention (guaranteed single invocation)
Prefix with `@agent-omb:` to guarantee a specific agent runs:
```
@agent-omb:critic review this architecture plan
@agent-omb:frontend-engineer implement the dashboard component
```
Best for: user-facing commands where a specific agent MUST run.

### 3. Agent Tool (programmatic, parallel batch)
Use the `Agent` tool with `subagent_type` matching the agent definition filename in `.claude/agents/omb/`:
```
Agent({
  subagent_type: "executor",  // matches .claude/agents/omb/executor.md
  model: "sonnet",            // override or omit to use agent default
  prompt: "...",
  description: "Implement auth middleware"
})
```
Best for: skill prompts that spawn multiple agents in parallel batches.

### Naming Convention
- `subagent_type` values MUST match the filename (without `.md`) in `.claude/agents/omb/`
- Valid: `executor`, `critic`, `frontend-engineer`, `db-specialist`
- Exception: `"general-purpose"` is a Claude Code built-in type (no agent definition file)
- Plugin agents use `omb:{name}` format in @-mention: `@agent-omb:executor`
</subagent_invocation>

Hook scripts live in `.claude/hooks/omb/`:
- `handle-session-start.sh` — SessionStart: session initialization
- `handle-user-prompt-submit.sh` — UserPromptSubmit: prompt preprocessing, keyword detection
- `handle-pre-tool.sh` — PreToolUse (Write|Edit|Bash): validate before tool execution
- `handle-post-tool.sh` — PostToolUse (Write|Edit): plan file naming validation — warns when plan files in `.omb/plans/` don't follow the session_id naming convention
- `handle-stop.sh` — Stop: persistent mode, completion checks
- `handle-pre-compact.sh` — PreCompact: state preservation before context compaction

Key lifecycle: SessionStart → UserPromptSubmit → PreToolUse → PostToolUse → Stop → SessionEnd
Exit code 2 = block action. Exit code 0 = proceed (stdout added as context for UserPromptSubmit/SessionStart).
Use `stop_hook_active` check to prevent infinite loops in Stop hooks.

Adding new hooks:
1. Create script in `.claude/hooks/omb/handle-{event}.sh`
2. Add entry to `.claude/settings.json` under the event key
3. Follow pattern in existing stubs (read stdin, process, exit 0 or 2)
</hook_system>

## Installation

**Install from PyPI:**
```bash
pip install oh-my-braincrew
# or: uv tool install oh-my-braincrew
```

**Install from source (development):**
```bash
git clone https://github.com/teddynote-lab/oh-my-braincrew.git
cd oh-my-braincrew
make setup
```

**Initialize in a project:**
```bash
cd /path/to/your/project
omb init
```

**Update to latest:**
```bash
pip install --upgrade oh-my-braincrew
# or: omb update
```

**Local development (plugin mode):**
```bash
claude --plugin-dir /path/to/oh-my-braincrew
```

**Release (maintainer):**
```bash
/omb release patch    # bump, build, tag, publish to PyPI
/omb release 1.2.3    # explicit version
```
